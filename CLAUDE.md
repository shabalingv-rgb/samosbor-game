## Project: Samosbor: Обратная сторона обоев (Godot 4)

**Genre:** Management Strategy, Real-Time Indirect Control Simulator, Psychological Horror  
**Analogues:** Plague Inc, Beholder, Cultist Simulator  
**Visual Style:** Hand-drawn on Soviet wallpaper texture (pencil/marker drawings with "Line Boil" effect)

A strategic management game where the player controls Samosbor—a mysterious entity born from (but not part of) Gigakhrushchyovka—attempting to infiltrate and consume a 100-floor tower block through psychological influence, resource management, cultist manipulation, and strategic biomass deployment.

**Team:** 2 people | **Timeline:** Flexible (spare time) | **Target:** PC

---

## Core Game Mechanics

### Two-Phase Cycle (per floor)

1. **Phase 1: Hidden Accumulation (Скрытой накопления)**
   - Duration: 5-20 minutes
   - Goal: Spread influence via ventilation, contaminate food, broadcast voices into dreams, create cultists, sabotage doors
   - Resources: Slowly generate Biomass + Terror/Psychosis
   - Risk: Detection increases with each action (Commandants can trigger inspection)
   - Win: Accumulate enough resources without exceeding 80% detection

2. **Phase 2: Active Samosbor (Самосбора)**
   - Triggered: Either forced by >95% detection OR player activates
   - Signal: Siren → hermetic doors seal
   - Goal: Consume isolated population (70%+) before Liquidators breach
   - Resources: Spend accumulated Biomass + Terror to manifest mutations, spawn creatures, fill floor with purple fog
   - Outcome: Success (floor consumed) | Partial (survivors fortified) | Failure (Liquidators breach, retreat)

### Two Resource Economy

**Biomass (Physical):** Obtained from consuming corpses, generated by cannibal mutations, from biological waste  
Uses: Spawning combat mutations, sealing/breaking doors, building nest defenses

**Terror/Psychosis (Mental):** Generated from sounds, visions, voices in dreams  
Uses: Creating cultists, paralyzing humans, forcing compliance, psychological spread

### Three Development Trees

1. **Mentality (Ментальность):** Psychological influence, stealth, cultist manipulation
2. **Biomass (Биомасса):** Physical mutations, combat creatures, structural damage
3. **Development (Развитие):** Nest protection, metabolic efficiency, entity evolution

### Two Game Modes
- **Story Mode:** Conquer 100 procedurally-generated floors. Floor 100 is the Liquidator HQ, built *around* the Core — not two separate endings but one: the Party fortified the one thing it could neither understand nor destroy, and called it a "headquarters" rather than name it truthfully. Five endings determined by the player's dominant skill tree, no dialogue choice screen — see `NARRATIVE_BIBLE.md` §3.
- **Infinite Mode:** Survive as Партия (Party) prepares thermobaric weapon (50-80 min survival)

## Game Premise & Lore

**Central Mystery:** Samosbor is an entity born FROM Gigakhrushchyovka but NOT part of it. Its goal: **consume the tower (replacing architecture with itself) and understand its own nature through expansion.**

### Core Elements

- **Samosbor:** A distributed consciousness of unknown origin. Not evil, but alien. Motivated by hunger + instinct + growing awareness.
- **Gigakhrushchyovka:** Infinite Soviet apartment building; Samosbor views it as territory to inhabit/replace.
- **Nest (Logovo):** Samosbor's core spawns in ventilation of a random floor. Must be protected or transported. If found and burned → game over.
- **Liquidators (Ликвидаторы):** Military cleanup crews sent by Партия (Party). Specialized units with unique tactics.
- **Cultists:** Humans with weak will who, under terror/psychic influence, become voluntary servants, opening doors, sabotaging systems, worshiping biomass fragments.

### Atmosphere

- **Visual:** Hand-drawn on Soviet wallpaper (pencil strokes, "living" line boil effect)
- **Tone:** Morbid, absurd, deeply strange. Samosbor's incomplete consciousness = incompleteness in art
- **Aesthetic:** Dread built from mundane Soviet detail (bland hallways, institutional silence), not jump scares

**Core wikis & lore:** [Samosbor Viki](https://samosbors8878.fandom.com/ru/wiki/), [Wikireality](https://wikireality.ru/wiki/%D0%A1%D0%B0%D0%BC%D0%BE%D1%81%D0%B1%D0%BE%D1%80)

## Architecture Notes

### Project Structure
```
scenes/
  ├── main/ (game loop, floor setup, camera)
  ├── npc/ (civilian, liquidator, official behavior)
  ├── player/ (Samosbor entity, influence fields, abilities)
  └── effects/ (purple fog, tentacles, mutations)

scripts/
  ├── game_manager.gd (global state, floor progression)
  ├── influence_system.gd (psychological spread, agent spawning)
  ├── skill_system.gd (three development trees)
  ├── procedural_generation/ (floor layout, NPC spawning)
  ├── ai/ (NPC pathfinding, Liquidator patrols, detection)
  └── data/ (skill data, floor presets, NPC templates)
```

### Key Design Principles
- **Modular floors:** Each floor is self-contained; procedural generation builds from reusable modules (residential, barracks, labs, storage)
- **Real-time simulation:** NPC schedules, Liquidator patrols, influence spread all happen continuously
- **Gating mechanics:** Player starts with detection risk; skill tree unlocks passive ability to hide, reduce detection, expand influence reach
- **Two-person workflow:** Clear separation between programming (core systems) and art (sprites, UI, effects, sound)

### GDScript Conventions (Godot 4)

- snake_case for variables/functions, PascalCase for classes
- Explicit types (`var resource: int = 10`, not `:=`)
- `@export`, `@onready`, `await` (Godot 4 syntax)
- Signals for decoupling (children emit up, parents listen down)
- One responsibility per script; keep <300 lines
- Disconnect in `_exit_tree()` for autoload signals

**Comments:** Only for WHY (non-obvious constraints, workarounds). Code should be self-documenting.

## Development Workflow

### Git/Version Control
- Feature branches for each major system (e.g., `feature/influence-system`, `feature/floor-generation`)
- Commit messages: "Add X, fix Y, refactor Z" — focus on what changed, not why (that goes in PR description)
- Merge to main only after playtesting key features

### Testing & Iteration
- Playtest frequently: Does the influence mechanic feel rewarding? Are Liquidators a credible threat?
- Adjust NPC detection ranges, Liquidator patrol speed, skill costs based on playtesting
- Use Godot's built-in debug tools: print() for quick checks, breakpoints for complex AI

### Asset Pipeline
- Sprites: Aseprite/Krita → PNG, commit to `assets/sprites/`
- Animations: AnimationPlayer in Godot, save as .anim
- Sound: Audacity → WAV/MP3, store in `assets/audio/`
- Data: Use `.gd` resource files or JSON for floor layouts, NPC templates, skill trees

## Known Constraints & Gotchas

1. **Procedural generation complexity:** Start with fixed floors, add randomization after core loop works
2. **NPC pathfinding in tight corridors:** Use simple A* or waypoint-based movement; full pathfinding can bottleneck with 100+ NPCs
3. **Influence spread can be expensive:** Use spatial hashing (quadtree) to avoid O(n²) detection checks per frame
4. **Godot 3 vs 4 differences:** This is Godot 4 only (no yield, use await; connect uses callable, not string signals)
5. **Mobile optimization:** 2D performance is good up to ~500 active nodes; beyond that, use object pooling and node culling

## Important for AI Collaboration

### Before Implementing
- Ask if unclear: Does this system belong in a single script or split into multiple nodes? Should this be a Signal or a direct function call?
- Propose architecture for complex systems (influence spread, procedural generation) before coding

### Code Review Priorities
- **Correctness:** Does NPC behavior match the design? Do Liquidators actually detect anomalies?
- **Performance:** For loops over large NPC arrays—use spatial queries, not brute force
- **Clarity:** Function names should make intent obvious (e.g., `apply_influence_to_npc()` vs `update_guy()`)

### Testing
- Unit tests for isolated systems (influence calculation, skill cost)
- Integration tests for floor generation (does every floor have at least one residential block?)
- Manual playtesting for feel (is the game fun? Do abilities feel powerful?)

## Communication & Decision-Making

- **Design decisions:** If a feature is ambiguous, propose two approaches with tradeoffs (not a survey of all options)
- **Scope creep:** Stick to the three development trees + two game modes for MVP; cosmetic features and story polish come later
- **Flexibility:** The Samosbor lore is intentionally vague; interpret creatively but stay true to the mood (bleak, absurd, unstoppable)
